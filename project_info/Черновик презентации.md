# Черновик плана презентации проекта Chart Detective

## Слайд 1. Приветственный

**Название проекта:** Chart Detective  
**Тема:** веб-игра по угадыванию страны по музыкальным чартам  

Кратко:
- приветствие;
- представление команды/автора;
- название проекта;
- общая идея: игрок слушает популярные треки и пытается определить, из какой страны этот музыкальный чарт.

Пример текста:
> Здравствуйте. Мы представляем проект Chart Detective. Это веб-игра, в которой пользователь слушает топовые треки из разных стран и пытается угадать страну по музыкальным предпочтениям.

## Слайд 2. Актуальность и идея проекта

Основная идея проекта:
- объединить музыку, географию и игровой формат;
- сделать интерактивное приложение, где пользователь не просто выбирает ответ, а анализирует треки, исполнителей и музыкальный стиль;
- показать работу с внешними API, базой данных, авторизацией и динамическим интерфейсом.

Почему это интересно:
- музыкальные чарты отличаются по странам;
- угадывание страны превращает прослушивание музыки в игру;
- проект позволяет применить сразу несколько технологий: backend, frontend, база данных, внешние сервисы.

## Слайд 3. Постановка задачи

Цель:
- разработать веб-приложение, в котором пользователь проходит игру из нескольких раундов и угадывает страны по музыкальным трекам.

Основные задачи:
- реализовать регистрацию и вход пользователей;
- получать музыкальные чарты для разных стран;
- показывать пользователю список треков и аудио;
- добавить выбор страны и таймер раунда;
- рассчитывать очки за правильные и частично близкие ответы;
- сохранять результаты игр и историю пользователя;
- оформить удобный интерфейс.

## Слайд 4. Общий сценарий работы

Как работает игра:
1. Пользователь заходит на сайт.
2. Может зарегистрироваться или играть без аккаунта.
3. Нажимает кнопку начала игры.
4. Система случайно выбирает несколько стран.
5. В каждом раунде загружается топ треков для одной страны.
6. Пользователь слушает музыку и выбирает страну.
7. Система начисляет очки.
8. После всех раундов показывается итоговый результат.

## Слайд 5. Стек технологий

Backend:
- Python;
- Flask;
- Flask-Login;
- SQLAlchemy;
- SQLite.

Frontend:
- HTML-шаблоны Jinja2;
- CSS;
- JavaScript;
- Tabler Icons;
- светлая и темная тема.

Внешние сервисы:
- SoundCharts API для получения музыкальных чартов;
- Apify для получения MP3-ссылок по Spotify-трекам.

Дополнительно:
- dotenv для хранения ключей API;
- requests для HTTP-запросов;
- локальный кэш аудиофайлов.

## Слайд 6. Архитектура проекта

Основные файлы:
- `app.py` - маршруты Flask, логика игры, работа с сессиями;
- `db.py` - подключение к базе данных;
- `db_models.py` - модели пользователя, игры и раунда;
- `get_chart.py` - работа с музыкальными API;
- `country_neighbors.py` - расчет очков по географической близости;
- `templates/` - HTML-шаблоны страниц;
- `static/` - CSS, JavaScript и кэш аудио;
- `data/` - JSON-файлы со странами, материками и соседями.

Идея архитектуры:
- Flask отвечает за маршруты и серверную логику;
- SQLAlchemy хранит пользователей, игры и раунды;
- JavaScript отвечает за динамику раунда: загрузку треков, таймер, поиск стран и выбор ответа;
- внешние API дают музыкальные данные.

## Слайд 7. База данных

В проекте используются три основные сущности:

**User**
- логин;
- никнейм;
- хэш пароля;
- количество сыгранных игр;
- лучший результат;
- дата регистрации.

**Game**
- пользователь, если игра привязана к аккаунту;
- итоговый счет;
- дата игры;
- статус завершения.

**Round**
- номер раунда;
- правильная страна;
- выбранная страна;
- очки за раунд;
- тип совпадения: точный ответ, соседняя страна, ошибка и т.д.

Зачем нужна база:
- хранить аккаунты;
- сохранять историю игр;
- показывать профиль пользователя и статистику.

## Слайд 8. Авторизация и профиль

Реализовано:
- регистрация пользователя;
- проверка уникальности логина и никнейма;
- хэширование пароля;
- вход и выход из аккаунта;
- защита страниц профиля;
- сохранение статистики пользователя.

Профиль показывает:
- никнейм и логин;
- дату регистрации;
- количество игр;
- лучший результат;
- средний счет;
- историю последних игр.

## Слайд 9. Игровая механика и начисление очков

В игре несколько раундов. В каждом раунде пользователь выбирает страну.

Система очков:
- точный ответ - 1000 очков;
- страна-сосед - 700 очков;
- страна через одного соседа - 400 очков;
- неправильный ответ или отсутствие ответа - 0 очков.

Идея:
- игрок получает частичные очки, если его ответ географически близок;
- это делает игру менее случайной и более справедливой;
- даже если страна не угадана точно, близкий ответ учитывается.

## Слайд 10. Работа с музыкальными данными

Для каждого раунда система:
- выбирает страну;
- получает музыкальный чарт через SoundCharts API;
- достает название трека, исполнителя, обложку и ссылку Spotify;
- через Apify пытается получить MP3;
- сохраняет аудио в локальный кэш;
- отдает подготовленные данные на фронтенд.

Зачем нужен кэш:
- чтобы не скачивать один и тот же аудиофайл повторно;
- чтобы ускорить загрузку раунда;
- чтобы уменьшить зависимость от внешних сервисов во время игры.

## Слайд 11. Интерфейс игры

На странице раунда есть:
- список топ-5 треков;
- аудиоплееры;
- обложки треков;
- таймер;
- текущий счет;
- поиск страны;
- группировка стран по регионам/материкам;
- кнопка подтверждения ответа.

Особенности:
- раунд начинается только после загрузки аудио;
- если время закончилось, форма отправляется автоматически;
- пользователь может быстро найти страну через поиск;
- интерфейс поддерживает темную и светлую тему.

## Слайд 12. Трудности при разработке

Основные трудности:
- получение стабильных аудиофайлов из внешних источников;
- ожидание полной загрузки аудио перед стартом таймера;
- работа с API-ключами и внешними сервисами;
- сопоставление названий стран с ISO-кодами и кодами музыкальных чартов;
- расчет частичных очков по соседним странам;
- сохранение состояния игры между страницами.

Как решали:
- добавили локальное кэширование аудио;
- вынесли страны, соседей и материки в JSON-файлы;
- сделали отдельный сервис `SoundChartsService`;
- используем Flask-сессии для текущей игры;
- сохраняем завершенные раунды в базе данных;
- добавили обработку ошибок, если треки не загрузились.
